# ユーザーストーリー

## 概要

採用・営業向けポートフォリオサイトのユーザーストーリーを INVEST 基準（Independent / Negotiable / Valuable / Estimable / Small / Testable）で記述する。各ストーリーは [システムユースケース](./system_usecase.md) と [テスト戦略](../design/test_strategy.md) の E2E ID（E01〜E12）に対応する。

## 規約

### ストーリー記述形式

```
US-NN: <一言タイトル>
**As a** <アクター>
**I want** <達成したいこと>
**So that** <得られる価値>

### 受入条件
- [ ] AC-NN-1: ...
- [ ] AC-NN-2: ...

### 関連
- UC: UC-NN
- E2E: E0N
- 画面: S0N
- リリース: v0.X
```

### 見積もり

ストーリーポイント（フィボナッチ）で見積もる。最初のイテレーションは「校正」を目的とし、過信しない。

| ポイント | 目安 |
|---|---|
| 1 | テキスト編集レベル |
| 2 | 単一コンポーネント追加 |
| 3 | 画面 1 つ |
| 5 | 画面 + データ統合 |
| 8 | 横断機能（全画面影響） |

## ストーリー詳細

### US-01: プロフィールを 30 秒で把握できる

**As a** 採用担当者
**I want** ホームを訪れて 30 秒で「誰がどんな専門領域か」を把握する
**So that** 面談に進める価値がある候補者か即座に判断できる

#### 受入条件

- [ ] AC-01-1: ホーム（`/`）の `<h1>` に氏名が表示される
- [ ] AC-01-2: 役職タイトル（例: Software Engineer）が `<h1>` の下に表示される
- [ ] AC-01-3: **キャッチコピー**（1 行、20〜40 文字程度）が表示される
- [ ] AC-01-4: **得意領域タグ**（5〜7 個、例: `Backend / DDD / TDD / AWS / Java`）がチップで表示される
- [ ] AC-01-5: **実績ハイライト**（1 行、例: 「経験 10 年 / 受託・自社プロダクトで 12 案件」）が表示される
- [ ] AC-01-6: Featured Works 3 件のカード（タイトル / サマリ / 技術タグ）がスクロールせずに見える、またはモバイルで 1〜2 スクロール以内
- [ ] AC-01-7: Skills Highlights（Backend / Frontend / Infrastructure）が表示される
- [ ] AC-01-8: 主要 CTA（「Works を見る」「Contact」）が表示される
- [ ] AC-01-9: フッターに SNS リンク（GitHub / LinkedIn / X）が配置される

#### 関連

- UC-01 / E01 / S01 / v0.1

---

### US-02: Works 一覧で実績の傾向を把握できる

**As a** 採用担当者
**I want** Works の一覧を見て、絞り込みながら実績の傾向を把握する
**So that** 自社の課題に関連する経験があるか判断できる

#### 受入条件

- [ ] AC-02-1: `/works/` で全 Works がカード形式で表示される
- [ ] AC-02-2: 各カードに「タイトル / 期間 / サマリ / 技術タグ / 詳細ボタン」が含まれる
- [ ] AC-02-3: 技術タグでフィルタできる（単一選択、URL に `?tag=...` を付与し共有可能）
- [ ] AC-02-4: 「All」で絞り込み解除
- [ ] AC-02-5: 0 件時は「該当する Work がありません + フィルタ解除」を表示
- [ ] AC-02-6: 件数表示（「N 件中 M 件を表示」）がフィルタ近傍にある
- [ ] AC-02-7: Tab で各タグへ移動、Enter で適用、現在選択は `aria-pressed="true"`

#### 関連

- UC-02 / E03 / S02 / v0.2

### US-03: Works 詳細で関与の深さと成果を判断できる

**As a** 採用担当者 / 業務委託発注検討者
**I want** Works 詳細を見て、課題から成果までのストーリーと関与の深さを判断する
**So that** 「再現性のある成果か」「同じ仕事を任せられるか」を評価できる

#### 受入条件

- [ ] AC-03-1: パンくず（Home > Works > Work タイトル）を表示
- [ ] AC-03-2: Work タイトル / 役職 / 期間 / カバー画像を表示
- [ ] AC-03-3: **「課題 → 挑戦 → 解決 → 成果」**の 4 ブロック構造で本文を表示
- [ ] AC-03-4: チーム規模（`team_size`）/ ポジション（`position`） / 関与の深さ（`involvement`） を表示
- [ ] AC-03-5: 業種（`domain`） / 機能領域（`category`）を表示
- [ ] AC-03-6: 使用技術タグを表示
- [ ] AC-03-7: 成果は定量指標（before/after）形式で表示
- [ ] AC-03-8: 外部リンク（GitHub Repo / Live Demo）が `target="_blank" rel="noopener noreferrer"` で別タブ
- [ ] AC-03-9: 「← 一覧に戻る」動線がある
- [ ] AC-03-10: 存在しない slug は 404（US-15 相当）

#### 関連

- UC-03 / E04 / S03 / v0.2

### US-04: Skills で技術領域の網羅性を確認できる

**As a** 採用担当者
**I want** Skills を見て、各カテゴリの経験年数・現役/過去・関連 Works を確認する
**So that** 求める技術領域に応えられるか評価できる

#### 受入条件

- [ ] AC-04-1: Backend / Frontend / Infrastructure / Practice の各カテゴリで一覧表示
- [ ] AC-04-2: 各 Skill に「経験年数」（`since` から自動計算）、「現役/過去」ステータスを表示
- [ ] AC-04-3: ★ レベル表現の場合は **凡例**を画面に明示（例: ★★★★★ = メンター可能）、または ★ を廃止し「現役/過去」のみ
- [ ] AC-04-4: 各 Skill から関連 Work へのリンク（Work 逆参照）を 1 つ以上表示
- [ ] AC-04-5: ハッシュ URL（`/skills/#java`）で該当 Skill にスクロールできる

#### 関連

- UC-04 / E05 / S04 / v0.3

### US-05: 稼働可否を確認して問い合わせ判断できる

**As a** 業務委託発注検討者
**I want** Contact 画面で稼働可否（availability）を確認する
**So that** 問い合わせに費やす時間が無駄にならないか判断できる

#### 受入条件

- [ ] AC-05-1: Contact 画面の冒頭に **稼働可否ステータス**（例: 「副業のみ可 / 週 2 日 / リモート」）を表示
- [ ] AC-05-2: 返信目標時間（「原則 2 営業日以内」）を表示
- [ ] AC-05-3: 相談可能な案件規模（任意）を表示

#### 関連

- UC-05 / E06 / S05 / v0.3

### US-07: ダークモードで快適に閲覧できる

**As a** 訪問者
**I want** ダークモードに切り替えて閲覧する
**So that** 夜間 / 画面が眩しい環境でも快適に読める

#### 受入条件

- [ ] AC-07-1: 初回訪問時は `prefers-color-scheme` を尊重
- [ ] AC-07-2: ヘッダーのトグルクリックで即時切替
- [ ] AC-07-3: `localStorage.theme` に永続化、リロードしても保持
- [ ] AC-07-4: ダークモードでも WCAG 2.1 AA のコントラストを満たす
- [ ] AC-07-5: View Transitions API 未対応ブラウザでは即時遷移（視覚的不連続なし）

#### 関連

- UC-12 / E08 / 全画面 / v0.3

### US-08: モバイルで快適に閲覧できる

**As a** 採用担当者（移動中・電車内）
**I want** モバイル端末で快適にポートフォリオを閲覧する
**So that** デスクに戻る前にスクリーニングを完了できる

#### 受入条件

- [ ] AC-08-1: iPhone SE（375px）以上で破綻しない
- [ ] AC-08-2: 768px 未満でハンバーガーメニューに切り替わる
- [ ] AC-08-3: タッチターゲットは 44×44 px 以上、間隔は 8px 以上
- [ ] AC-08-4: 画像は AVIF/WebP 優先、LCP 候補は < 200KB
- [ ] AC-08-5: モバイルでホームの主要セクションが 6〜8 スクロール以内

#### 関連

- UC-13 / E09 / 全画面 / v0.3

### US-09: 検索エンジンに正しく索引される

**As a** クローラ（および検索流入を期待するサイトオーナー）
**I want** 主要ページが検索エンジンに索引される
**So that** 採用担当者が検索でこのサイトに到達できる

#### 受入条件

- [ ] AC-09-1: `/sitemap.xml` が自動生成される
- [ ] AC-09-2: `/robots.txt` が production では索引許可、staging では `Disallow: /`
- [ ] AC-09-3: 各ページに `og:title` / `og:image` / `og:url` / `og:description` が設定される
- [ ] AC-09-4: `/docs/*` には `<meta name="robots" content="noindex,nofollow">` が設定される（[ADR-0003](../adr/0003-mkdocs-coexistence-strategy.md)）

#### 関連

- UC-11 / E10 / 全画面 / v0.1

### US-10: キーボード / スクリーンリーダーで全機能にアクセスできる

**As a** 多様な訪問者（キーボード操作者、スクリーンリーダー利用者）
**I want** マウスを使わずに全コンテンツに到達する
**So that** アクセシビリティ要件を満たし、誰でもポートフォリオを評価できる

#### 受入条件

- [ ] AC-10-1: Tab で全インタラクティブ要素にフォーカスできる
- [ ] AC-10-2: フォーカスは `:focus-visible` でリング表示される
- [ ] AC-10-3: スキップリンク（"メインへ移動"）が存在する
- [ ] AC-10-4: `<header> / <nav> / <main> / <footer>` のランドマークが配置される
- [ ] AC-10-5: ハンバーガーメニュー展開時はフォーカストラップ + Esc で閉じる + body スクロール抑止
- [ ] AC-10-6: axe-core via Playwright で violations 0
- [ ] AC-10-7: NVDA / VoiceOver で主要画面が読み上げ可能（リリース前手動検証）

#### 関連

- UC-14 / E11 / 全画面 / v1.0

### US-11: Tech Notes から技術的詳細に到達できる

**As a** 同業エンジニア
**I want** ポートフォリオから Tech Notes（MkDocs）へ遷移し、ADR や設計ドキュメントを読む
**So that** 技術的判断の根拠と思考プロセスを参考にできる

#### 受入条件

- [ ] AC-11-1: ヘッダーナビに「Tech Notes」（「Docs」ではない）と「↗」アイコン付きで表示
- [ ] AC-11-2: クリックで `/docs/` に同一タブ遷移
- [ ] AC-11-3: `/docs/` トップに「これは個人の学習・設計メモです」のガイダンスバナー
- [ ] AC-11-4: MkDocs 側のヘッダーに「← ポートフォリオに戻る」リンク
- [ ] AC-11-5: ファビコン・サイト名・配色トークンが Astro と共通

#### 関連

- UC-07 / E02 / S91 / v1.0

### US-12: SNS シェアで OGP プレビューが正しく表示される

**As a** 訪問者（シェアする側）
**I want** ポートフォリオの URL を SNS / Slack / Teams で共有したときに、適切なプレビュー画像とタイトルが表示される
**So that** 受け取った人にコンテンツの第一印象を渡せる

#### 受入条件

- [ ] AC-12-1: ホーム / Works 一覧 / Works 詳細 / Skills / Contact ですべて OGP メタタグが出力される
- [ ] AC-12-2: OGP 画像が 1200×630 で `og:image` が `Content-Type: image/png` または `image/jpeg`
- [ ] AC-12-3: Works 詳細では `og:title` が Work タイトル + 期間
- [ ] AC-12-4: Twitter Card が `summary_large_image` で出力される

#### 関連

- UC-11 / E10 / 全画面 / v1.0

### US-13: Markdown 編集で公開できる（オーナー）

**As a** サイトオーナー
**I want** Works / Skills / Profile を Markdown フロントマターで編集し、Git push だけで公開する
**So that** CMS を持たずに低コストで持続的に更新できる

#### 受入条件

- [ ] AC-13-1: `apps/web/src/content/` 配下の Markdown 編集だけで Works / Skills / Profile を更新できる
- [ ] AC-13-2: フロントマターのスキーマ違反は `astro check` で CI 失敗となる
- [ ] AC-13-3: 「コンテンツのみの変更」では Lighthouse 段階予算で警告のみ（リリースを止めない）
- [ ] AC-13-4: PR テンプレートに「コンテンツ変更」のチェック項目を含む

#### 関連

- UC-08 / （CI 検証） / - / v0.1

---

### US-14: 障害時に 1 時間以内で復旧できる（オーナー）

**As a** サイトオーナー
**I want** 障害発生時に runbook に沿って 1 時間以内に復旧する
**So that** SLO 99.5% を維持し、特に採用面接前後の停止リスクを最小化する

#### 受入条件

- [ ] AC-14-1: UptimeRobot が `/healthz` を 5 分間隔で監視し、連続 2 回失敗で通知（Pushover + メール）
- [ ] AC-14-2: `ops/runbook/rollback.md` に Slug ロールバック手順がコマンド単位で記載
- [ ] AC-14-3: `ops/runbook/disaster-recovery.md` に Cloudflare DNS 切替で GitHub Pages へ退避する手順
- [ ] AC-14-4: `ops/runbook/pre-interview-freeze.md` に「面接 2 営業日前から hotfix のみ」のルール
- [ ] AC-14-5: 月次で staging に対し Slug ロールバックを実演する手順

#### 関連

- UC-09, UC-10 / （runbook） / - / v0.1
