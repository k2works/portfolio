## 課題

10 年以上稼働するインターネット通販の販売管理システム（受注 / 在庫 / 出荷 / 顧客対応）が Rails 3.2（当時の最新は Rails 7 系）+ Ruby 2.x + MySQL 5.x という大幅にレガシーな構成に取り残されていた。本番運用は安定していたものの、新規機能追加のたびに手動デプロイで数時間かかり、再現可能なビルド環境も存在せず、属人的な運用になっていた。通販事業のキャンペーン・セール対応で短納期の機能追加が多発するため、安全な変更プロセスの確立が急務だった。

## 挑戦

業務影響を最小化しながら、レガシースタックの上で **継続的な機能追加と運用近代化** を両立する挑戦。受注・在庫・出荷フローはビジネスのコアであり、停止 = 売上損失に直結する。クローズドなプロダクトのため公開チャネルでの情報共有はできず、社内ドキュメントと自動化スクリプトに知識を集約する必要があった。組織体制の変化やビジネス要件の変動を吸収しながら、長期にわたるシステムの「使える状態」を維持する。

## 解決

Docker / docker-compose で開発環境を再現可能化し、AWS CodeBuild + CodeDeploy で staging / production の自動デプロイパイプラインを構築。ステージング用の `buildspec-stg.yml` と本番用 `appspec.yml` を分離し、`Dockerfile-base` に共通基盤を切り出して保守性を改善した。AsciiDoc ベースの内部ドキュメント（`docs/`）で要件・運用手順を一元化し、機能追加（メニュー絞り込み・受注フロー改修等）は Cucumber（`features/`）+ RSpec（`spec/`）の二段テストで品質を担保した。
